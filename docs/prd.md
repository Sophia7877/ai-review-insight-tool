## 3. 需求背景

品牌和电商运营团队在新品上市、大促复盘、日常评论监测和内容选题时，会接触大量用户评论。传统处理方式通常依赖人工逐条阅读、手动归类和经验判断，效率低，且难以形成可复用的运营动作。

本项目希望用 AI 将用户评论转化为结构化洞察，帮助运营人员快速判断：

- 用户为什么选择产品？
- 用户为什么犹豫或放弃？
- 哪些卖点应该被放大？
- 哪些疑虑需要在详情页、客服 FAQ 或内容中提前解释？
- 如何将评论反馈转化为运营动作和复盘指标？

## 4. 需求目标

### 4.1 业务目标

帮助运营团队把用户评论转化为可执行的内容选题、商品详情页优化建议、客服 FAQ 和活动复盘建议，提高用户对产品价值的理解、信任和购买意愿。

### 4.2 用户目标

让运营人员通过复制评论、上传文件或加载样例，在较短时间内获得一份结构化评论洞察报告。

### 4.3 产品目标

建立一条轻量级评论分析工作流：

```text
评论输入 → AI 分析 → 人工复核 → 洞察采纳 → 运营动作 → 指标复盘
```

## 5. 成功指标

| 指标 | 目标值 | 说明 |
| --- | --- | --- |
| 评论分析完成率 | 80%+ | 用户输入评论后能成功生成报告 |
| 洞察采纳率 | 30%+ | 输出洞察被写入周报、Brief 或复盘 |
| 内容选题转化率 | 20%+ | 输出选题进入内容排期 |
| 报告复制/下载率 | 25%+ | 用户复制总结或下载报告/数据 |
| 人工修正率 | 低于 40% | AI 输出需要大幅修改的比例 |
| 7 日复用率 | 25%+ | 用户一周内再次使用工具 |

## 6. 目标用户与使用场景

### 6.2 核心场景

## 8. 用户流程

### 8.1 主流程

```text
进入 Demo 页面
  ↓
选择运营场景模板
  ↓
加载样例评论 / 粘贴评论 / 上传文件
  ↓
点击“生成洞察”
  ↓
查看情绪分布、高频主题、痛点、选题和运营建议
  ↓
复制周报总结 / 下载 Markdown 报告 / 下载 JSON 数据
  ↓
将结果用于内容排期、详情页优化、客服 FAQ 或活动复盘
```

### 8.2 异常流程

| 异常情况 | 处理方式 |
| --- | --- |
| 用户未输入评论 | 提示用户加载样例或输入评论 |
| 上传文件为空 | 不生成报告，保留当前页面状态 |
| CSV 无评论字段 | 尝试按纯文本逐行读取 |
| AI 输出不准确 | 通过人工复核标准判断是否采纳、修改或弃用 |

## 10. 页面与交互说明

### 10.1 首屏展示

首屏展示产品定位、核心价值、评论样本数、高频主题数、选题建议数和洞察可采纳分。

### 10.2 场景模板区

展示三个可点击模板，帮助用户快速理解产品可以应用在哪些运营场景。

### 10.3 运营转化区

说明用户评论如何转化为购买理由、流失阻碍、内容入口和运营动作。

### 10.4 输入与输出区

左侧为评论输入区，右侧为运营洞察看板，包括情绪分布、高频主题、痛点、内容选题、购买理由和复盘指标。

## 11. 数据结构

### 11.1 输入数据

| 字段 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| review_id | string | 否 | 评论 ID |
| product_name | string | 否 | 商品名称 |
| rating | number | 否 | 用户评分 |
| review_text | string | 是 | 评论正文 |

### 11.2 输出数据

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| sample_size | number | 评论数量 |
| sentiment | object | 正面、中性、负面数量 |
| purchase_motivations | array | 购买动机 |
| conversion_barriers | array | 流失阻碍 |
| representative_pain_points | array | 代表性痛点 |
| content_topics | array | 内容选题 |
| selling_points | array | 可放大卖点 |
| metrics | array | 复盘指标 |
| operation_advice | string | 运营建议 |

## 13. 非功能需求

| 类型 | 要求 |
| --- | --- |
| 易用性 | 用户无需注册、无需接入后台即可体验 |
| 可读性 | 输出结果应面向运营人员，而不是技术人员 |
| 响应速度 | Demo 样例分析应在 3 秒内完成 |
| 可扩展性 | 后续可接入大模型 API 和多行业标签体系 |
| 数据安全 | Demo 不使用真实用户隐私数据 |
| 兼容性 | 页面需适配桌面端和移动端浏览 |

## 14. 风险与应对

| 风险 | 影响 | 应对方式 |
| --- | --- | --- |
| AI 输出过于泛化 | 用户不愿采纳 | 增加人工复核标准和行业 Prompt |
| 样例数据代表性不足 | 洞察可信度有限 | 后续补充多行业样例 |
| 用户不愿上传数据 | 影响真实使用 | 提供复制粘贴和脱敏样例 |
| 没有平台数据接入 | 留存可能受限 | V0.2 后考虑导入更多格式 |
| 结果难以进入工作流 | 工具价值下降 | 提供 Markdown、JSON 和复制能力 |
